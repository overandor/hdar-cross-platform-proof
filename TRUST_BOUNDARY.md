# HDAR Trust Boundary — Open Validation Boundaries and Attestation Roadmap

This document records what the current HDAR proof authenticates, what it does **not** authenticate, and the exact path to closing each open boundary. It exists so that no claim is overstated and so that the next engineer can resume the work without reconstructing the reasoning.

## Current authentication state

| Area                                 | Assessment |
| ------------------------------------ | ---------: |
| Workspace content integrity          | A          |
| Host A authentication                | A          |
| E1→E2 lineage mechanics              | A−         |
| Bounded deterministic replay         | A−         |
| Cross-runtime execution evidence     | B+         |

## What the verifier authenticates well

- E1 manifest integrity (SHA-256 over canonical JSON)
- Ed25519 owner signature on E1 (Host A identity)
- Workspace hashes (content-addressed blocks)
- Parent manifest linkage (E2.parent == E1.manifest_hash)
- Preserved shared files between epochs
- Bounded deterministic replay (5-stage pipeline output hash)

## What the verifier does NOT authenticate

### 1. Host B origin (the largest gap)

Fields such as `hostname`, `platform` string, `machine_nonce`, timestamps, and `platforms_differ` are generated by the Host B program. They are evidence **inside** the report, not external proof of the report's origin. A modified runner can fabricate them while preserving internal consistency.

**Status:** Infrastructure built — pending real workflow run.

**What's built:**
- `.github/workflows/reproduction_matrix.yml` creates a single evidence archive per matrix job containing: host_b_report.json, E2 capsule, workflow_identity.json, source_commit.txt, challenge_nonce.txt. A single `actions/attest@v4` call attests the archive, binding all evidence to the workflow run, commit, and GitHub OIDC identity.
- `verify_attestations.py` verifies these attestations using `gh attestation verify`, with `--archive` flag for evidence archives and `--report`/`--e2-tar` for legacy per-file attestations.
- The workflow's `verify-matrix` job runs `verify_attestations.py` automatically after all Host B jobs complete.
- SLSA classification: signed provenance from a hosted platform = Build Level 2. Level 3 requires a hardened build platform with stronger isolation.

**What's pending:**
- A real workflow run to generate actual attestations at the current commit.
- Verification that the attestations bind the correct commit and repository.
- Documentation of the attestation verification results.

**Trust chain (per web verdict):**
```
Owner signature → authenticates E1 and the owner
Challenge-bound Host B signature → authenticates the continuation report inside the job
GitHub artifact attestation → authenticates workflow, repository, commit, and artifact digest
Independent verifier → authenticates the entire E1 → execution → E2 transition
```

**Roadmap (remaining):**
- Have Host B sign reports with an ephemeral key whose certificate chains to a provider-attested identity.
- Record challenge-bound nonces: the verifier issues a challenge before Host B runs; Host B must include the challenge in its signed report. (The workflow already passes `--challenge-nonce` with the run ID.)
- Hardware-level attestation would require a trusted execution environment, TPM-backed identity, or confidential-computing evidence.

### 2. Provider-backed provenance

The published evidence establishes four recorded Host B runtime configurations across one Codespaces environment and three separately provisioned GitHub-hosted Actions jobs — not four independent cloud providers. GitHub's public runner documentation supports separately provisioned hosted environments but does not give this repository evidence that the three Actions jobs used an independently identifiable cloud provider distinct from Codespaces.

**Status:** Open — infrastructure built, pending real workflow run with attestations.

**What's built:**
- The reproduction matrix workflow attests each Host B evidence archive with GitHub artifact attestations (Sigstore-signed, transparency-logged).
- This authenticates: repository, organization, workflow, commit SHA, triggering event, GitHub Actions OIDC identity, and artifact digest.
- This is provider-backed **workflow execution provenance**, not hardware attestation. Hardware-level attestation would require a TEE, TPM-backed identity, or confidential-computing evidence.

**Roadmap:**
- Dispatch the canonical release bundle (see `release.py`) to genuinely independent providers: E2B, Google Colab, ChatGPT Linux, an independently operated VPS, and an unaffiliated third party.
- For each provider, capture provider-attested execution evidence (GitHub artifact attestations, E2B sandbox metadata, Colab session info).
- Publish a reproduction matrix showing the same E1 manifest hash across all providers.

### 3. Verifier implementation independence

The Python verifier (`verify_all.py`) shares Python with the sealer (`host_a_seal.py`). It re-implements all checks from the capsule format spec, but a single implementation author wrote both. The independent Rust verifier closes this gap.

**Status:** Infrastructure built — independent Rust verifier + CI integration.

**What's built:**
- `rust_verifier/` — a complete independent verifier in Rust that reimplements all 21 checks from the capsule format specification. Shares NO code with the Python verifier. Uses independent cryptographic libraries: `sha2` (RustCrypto), `ed25519-dalek`, `serde_json`.
- All 4 published platforms pass 21/21 with the Rust verifier (84/84 total).
- The Rust verifier handles backward compatibility with older evidence formats (field name variations, absent source-commit binding).
- `prove.py` runs both verifiers and requires both to pass.
- `.github/workflows/verify.yml` runs both verifiers + failure-injection tests in CI on every push/PR.

**Roadmap (remaining):**
- Have an unaffiliated reviewer audit the verifier logic.

### 4. Adversarial-host resistance

The current proof assumes a cooperative Host B. A malicious Host B could:
- Fabricate platform strings and nonces (see boundary 1).
- Run a modified pipeline and report a fabricated output hash.
- Sign a fraudulent E2 with an ephemeral key (the verifier checks the signature is valid, not that the signer is trustworthy).

**Status:** Partially addressed — failure-injection tests prove the verifier rejects tampered evidence.

**What's built:**
- `test_failure_injection.py`: 10/10 tampered evidence correctly rejected, covering corrupt manifests, signatures, content blocks, lineage, platform strings, receipts, output hashes, and owner key swaps.
- Pipeline output hash is recomputed from E2 workspace content (not just checked in report).
- `.github/workflows/verify.yml` runs failure-injection tests in CI on every push/PR.

**Roadmap (remaining):**
- Bind the Host B report to provider-attested execution (closes the fabrication gap — infrastructure built, pending real workflow run).
- Add more adversarial tests: modified runner, swapped E2 from different E1, replayed evidence with stale nonces.

### 5. Live agent/process continuity

The proof preserves **files**, not live cognition. The agent state is a JSON file; the "continuation" is a deterministic pipeline execution, not a running agent that resumed its thought process.

**Status:** Not demonstrated. Not a current goal of this proof.

**Roadmap (if pursued):**
- Define a formal agent-state serialization that captures working memory, not just task state.
- Prove that a restored agent produces the same outputs as an uninterrupted agent for a defined task class.
- This is a research problem, not an engineering fix.

### 6. Production key governance

The owner private key is in `.gitignore` and was generated on Host A. There is no documented key rotation, key escrow, or key recovery procedure.

**Status:** Not demonstrated.

**Roadmap:**
- Document key generation, storage, rotation, and revocation procedures.
- Use a hardware-backed key (HSM, Secure Enclave) for production seals.
- Document the recovery path if the owner key is lost.

## The strengthened chain (target state)

```
Host A Ed25519 signature
    authenticates owner and E1

GitHub workflow attestation
    authenticates repository, commit and workflow execution

Host B ephemeral signing key
    signs report, E2 manifest and termination receipt

Artifact digest
    binds report, capsule and execution outputs

Independent verifier (Rust or Go)
    verifies every identity and transition
```

GitHub artifact attestations establish where and how an artifact was produced and bind provenance to the artifact digest. GitHub supports verification of these attestations, including offline verification using downloaded attestation bundles and trusted roots. GitHub OIDC is useful for federated workflow identity, while artifact attestations provide the more direct build-output provenance object.

## Important version mismatch

The public successful workflow ran at commit `21a6302`. The repository now shows additional commits beyond that run. Therefore, the existing Actions evidence proves the workflow state at `21a6302`, not automatically every later source change.

A strong verifier should explicitly enforce:

```
evidence.source_commit
    =
attested workflow commit
    =
reviewed protocol commit
```

Any newer source-commit binding, termination receipt changes, or audit additions need a new successful public run at the current commit.

## Strongest claim that survives scrutiny

> **HDAR demonstrates that an owner-signed, content-addressed workspace can be transported to separately provisioned heterogeneous runtimes, restored byte-exactly, deterministically advanced through a bounded task, and resealed into a cryptographically linked successor epoch. The current proof authenticates Host A and artifact lineage, with implementation-independent verification confirmed by two independent codebases (Python and Rust, both must pass). Externally attested Host B provenance and adversarial-host resistance remain open validation boundaries.**

## Canonical release bundle (the next milestone)

The `release.py` module freezes one canonical E1, owner keypair, embedded runner, verifier, and dependency lock into a single immutable `release_bundle.tar.gz`. Multiple Host B providers can dispatch this identical bundle to prove they continued the **same** E1, not a rebuilt one.

```bash
# Build once
python3 release.py build --out /tmp/hdar_release

# Dispatch to multiple providers (identical E1)
python3 run_proof.py --reuse-release /tmp/hdar_release/release_bundle.tar.gz --provider e2b
python3 run_proof.py --reuse-release /tmp/hdar_release/release_bundle.tar.gz --provider codespaces
python3 run_proof.py --reuse-release /tmp/hdar_release/release_bundle.tar.gz --provider colab
python3 run_proof.py --reuse-release /tmp/hdar_release/release_bundle.tar.gz --provider chatgpt-linux
```

Every run must begin with the same E1 manifest, owner key, runner hash, and verifier hash. Different Host B keys, E2 hashes, timestamps, and provider identities are expected. **Different E1 hashes mean it is a different experiment.**
